### Q1: RF-06（一覧表示順の保証）はどこで実現しますか？（申し送り #1。あわせて tie-break の第 2 ソートキーを確定します）

a) CMP-002（Todo Backend API）— API がソート済み配列を返し、順序保証をコントラクトの一部にする
b) CMP-001（Todo Frontend）— 表示時にソートし、API は順序未保証のまま
c) 両方 — API ソート + フロント防衛的ソート
d) Other

**Trade Offs:** a) は順序が「一覧」という能力の意味論として API 仕様に載り、E2E（RF-02）や将来のクライアントが安定する。永続化層の Scan はどのみち順序不定なので、どこかで必ずメモリ内ソートが要る — 実装コストは a/b 同等。順序保証の追加はステータス・形状を変えないため Q2=b（後方互換）の範囲内。b) は API 仕様上「順序未保証」が残り、BT-2 の検証が表示層経由でしかできない。c) は同一関心の二重実装で、二重定義の解消（RF-12 と同じ思想）に逆行するドリフト源。

**Recommendation:** a — 業務ルールの唯一の強制点を CMP-002 に置く本システムの原則（components.yaml の境界定義）と整合する。第 2 ソートキーは **id 降順** で確定したい（requirements.md RF-06 の指名候補。ULID は辞書順＝時系列なので createdAt 降順と単調整合し、追加属性なしで決定性を得られる）。

[Answer]: a — API（CMP-002）がソート済み配列を返し、順序保証をコントラクトに載せる。第 2 ソートキーは id 降順で確定。

### Q2: RF-12（CORS の一元化）の集約先はどれにしますか？（申し送り #2）

a) 両方撤去（0 箇所）— 本番は CloudFront 同一オリジン、ローカルは Vite proxy 同一オリジンのため CORS 自体が不要
b) Hono `cors()` に一元化（API Gateway `corsPreflight` を撤去）
c) API Gateway `corsPreflight` に一元化（Hono `cors()` を撤去）
d) Other

**Trade Offs:** a) は最小構成で「同一オリジン前提」という設計意図が最も明確（A-1 / AR-O10 と一致。CORS ヘッダ消失は BP-1 許容変更 5 で承認済み）。リスクは将来ブラウザから異オリジンで直接叩く経路が生まれた場合だが、既知経路（CloudFront / Vite proxy / docker-compose）はすべて同一オリジン。b) は proxy を外した直叩き開発の柔軟性が残るが、使われない許可オリジン設定の保守が続く。c) はインフラ層に寄るため、API Gateway が存在しない docker-compose ローカルと本番で挙動差が生まれ、E2E の証明力が落ちる。

**Recommendation:** a（両方撤去）— RF-12 受入基準は「高々 1 箇所（0 箇所も可）」であり要件上も許容済み。E2E（RF-02）が本番相当経路とローカル経路の全 BT 動作を pass/fail で検証することを条件に、最も単純な形を採る。

[Answer]: a — 両方撤去（0 箇所）。同一オリジン前提を設計意図として明確化。E2E（RF-02）による全 BT 検証を撤去の条件とする。

### Q3: RF-03 のテストコード内の境界値リテラル（200 / 1000）はどう扱いますか？（申し送り #5）

a) テストはリテラル維持 — 境界値をテストに直書きし、共有定数の変更を検知する独立した検証点として機能させる
b) テストも共有定数から導出 — `MAX_TITLE_LENGTH + 1` のような定数演算で生成し、リテラルを 0 箇所にする
c) ハイブリッド — コントラクト検証テスト（schema 境界値）のみリテラル、その他のテストデータは定数導出
d) Other

**Trade Offs:** a) は誰かが共有定数を誤って変えるとテストが落ちる＝コントラクト変更に意図的なレビューを強制できる。重複は残るがそれ自体が目的。b) は DRY だが、定数変更にテストが自動追従してしまいコントラクト変更の検知装置として無力化する（トートロジー化）。c) は両立できるが「どちらに書くか」の規約説明コストが増える。

**Recommendation:** a — RF-03 受入基準が「プロダクションコード内」にスコープ限定された経緯（requirements.md Refine 記録 L-1）の趣旨は、テストを独立した検証点として残すこと。教材としても「テストの境界値はあえて直書きする」判断の記録に価値がある。

[Answer]: a — テストはリテラル維持。共有定数の誤変更を検知する独立検証点として機能させる。

### Q4: CMP-003（Shared Contract）の公開面 — frontend へ zod schema も公開しますか？（申し送り #7 設計観点・前半）

a) 既決どおり: frontend は型と制約定数のみ import（schema は CMP-002 専用の公開面とする）
b) schema も frontend へ公開し、フォーム入力補助（クライアント側プレ検証）に再利用する
d) Other

**Trade Offs:** a) は「CMP-001 は防衛線ではない／検証の強制点は CMP-002」という components.yaml の境界が構造的に保たれ、検証ライブラリが frontend バンドルに入らない。b) は入力補助メッセージと検証境界の一貫性が上がるが、RF-03 本文「frontend は型と定数を import」の既決と乖離するため要件著者との整合確認が必要（units.md 申し送り #7 の注意書き）。検証が 2 箇所で走ることで「強制点はどこか」が曖昧になる。

**Recommendation:** a — 要件の既決を維持。frontend の文字数補助は共有定数（200/1000）参照で十分であり、境界の明確さ（Simplicity is a feature）を優先する。

[Answer]: a — 既決どおり frontend は型と制約定数のみ import。schema は CMP-002 専用の公開面とする。

### Q5: CMP-003 のビルド形態はどうしますか？（申し送り #7 設計観点・後半。Q4 と関連）

a) ビルドなし（ソース直接参照）— パッケージの公開面が TypeScript ソースを直接指し、消費側ツールチェーン（Vite / Vitest / Lambda バンドラ）が直接取り込む
b) 事前ビルド（tsc で dist + 型宣言を生成）— 消費側は成果物を参照する
c) 本ステージでは「CMP-003 が独立に型検査・テスト可能であること」のみを制約とし、具体形態は code-generation へ委譲
d) Other

**Trade Offs:** a) はビルドステップ・成果物管理・watch ビルドが不要で最小。既存の消費側ツールはすべて TS ソースを直接消費できる。unit-dependencies.md の「CMP-003 を build 順序の先頭に置く」制約は型検査順序として自明に満たされる。制約は消費側が TS を解釈できるツールチェーンに限られること（本リポジトリでは全消費者が該当）。b) はパッケージ境界が成果物として明確でツールチェーン非依存だが、開発時の watch ビルドと dist 管理が常時必要になり、4 パッケージ教材の複雑性が増す。c) はステージ境界の純粋性はあるが、申し送り #7 が「functional-design へ」と明示委譲した経緯と齟齬が出る。

**Recommendation:** a — 定義専用・実行時振る舞いなし（components.yaml）という CMP-003 の性格に最も適合し、教材の簡潔性を保つ。型検査は CI（RF-01）の全 workspace `tsc --noEmit` が担保する。

[Answer]: a — ビルドなし（ソース直接参照）。型検査は CI（RF-01）の全 workspace tsc --noEmit が担保する。
