## 5. BP-1 許容変更 4・5 の Q2=b 適合性判断 — 条件付き合格（人間への確認を推奨）

refine（H-3 / M-1 対応）で追加された許容変更 2 件は、**人間が Q2=b で明示的に承認した 4 件のリストには含まれていない**。reviewer としての判断:

**許容変更 4（RF-07: 不正ボディ かつ 不存在 id の複合ケースで 404→400）— Q2=b の趣旨内と判断する。**
- 人間が Q2=b で明示承認した「条件付き書込によるアトミック化」の**論理的帰結**である。条件付き書込では不正ボディから書込式を構成できないため、検証先行（400 優先）は実質的に不可避。別途存在確認を挟めば 404 優先を保てるが、それはアトミック化（DynamoDB 呼出 1 回化）の承認意図そのものを無効化する
- 400/404 とも既存コントラクトに定義済みのステータスであり、複合ケースの優先順位はコントラクト未規定。「後方互換を保てる範囲」という Q2=b の文言に反しない
- 影響はエッジケース（不正ボディ + 不存在 id の同時成立）に限定され、要件本文で期待値が pass/fail テストとして確定されている

**許容変更 5（RF-12: CORS レスポンスヘッダの変更・消失）— A-1 を前提とすれば Q2=b の趣旨内と判断する。ただし前提依存。**
- コントラクト維持対象（api-documentation.md の System Contracts 表: メソッド/パス/ステータス/レスポンス形状）に CORS ヘッダは含まれず、既知クライアント（同梱 SPA・同一オリジン、AR-O10 の CSP が補強）には観測不能
- ただしこの判断は **A-1（外部クライアント不存在）という検証フラグ付き仮定に立脚**している。仮定が崩れた場合の再評価対象に RF-12 が明記されている点は適切

**結論**: 両件とも (a) 承認済み決定の派生的帰結、(b) 影響範囲が狭く要件で確定済み、(c) 隠蔽せず BP-1 の閉じたリストに理由付きで明記、という 3 条件を満たすため、**not ready の根拠とはしない**。一方で、振る舞い保持を主軸とする本 intent において「外部から観測可能な振る舞い変更の許容リスト」は人間の専権事項に最も近い領域であり、エージェント側の解釈で確定させたまま設計へ進むべきではない。**次の人間チェックポイントで許容変更 4・5（および前提 A-1）の明示承認（acknowledge）を取得すること**をオーケストレーターへ推奨する。承認が得られない場合、RF-07 は「アトミック化の撤回（OOS 化）」か「存在確認の併用（404 優先維持・アトミック性は条件付き書込で担保）」の再設計、RF-12 は「Hono 側単独残置」への限定が代替案となる。
