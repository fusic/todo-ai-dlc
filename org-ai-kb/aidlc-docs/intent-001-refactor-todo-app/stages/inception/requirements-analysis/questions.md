### Q1: v1 ドリフト 4 件の扱い — 仕様に合わせて実装を直すか、実装を正として仕様を更新するか

対象: BO-O2（FR-002 の作成日時が UI 未表示）/ BO-O3（TodoForm が設計記述と異なり作成専用）/ AR-O3（IAM が v1 設計の 5 アクション限定より広い）/ AR-O4（SECURITY-03 構造化ログが未実装）

a) すべて v1 仕様・設計を正とし、実装を仕様に合わせて修正する
b) すべて実装を正とし、v1 仕様・設計記述の側を現状に合わせて更新する
c) 件別に判断する: BO-O2 は実装修正（createdAt 表示を追加）、BO-O3 はドキュメント更新（現実装の UX は妥当）、AR-O3 は実装修正（最小権限化）、AR-O4 は実装修正（構造化ログ導入）
d) その他（件別の指定があれば [Answer] に記載）

**Trade Offs:** (a) は要件→実装の一貫性が最も強く教材的に明快だが、BO-O3 のように現実装の方が UX 上妥当なケースまで巻き戻すコストが発生する。(b) は実装コストゼロだが、IAM 過剰権限（AR-O3）やログ未実装（AR-O4）を「仕様」として追認することになり、品質向上という本 intent の主軸に反する。(c) はドリフトの性質（ドキュメントの陳腐化 vs 実装の不足）に応じて最適化でき、RE 成果物の提案（BO-P1 / BO-P2）とも整合する。

**Recommendation:** **c**。ドリフトは一律に扱える性質ではない。BO-O3 のみ「実装が正」（設計記述の更新）、残り 3 件は「仕様が正」（実装修正）とするのが、振る舞いを保ちつつ品質を上げる intent に最も合致する。

[Answer]: c — 件別判断。BO-O2 は実装修正（createdAt 表示追加）、BO-O3 はドキュメント更新（現実装の UX を正とする）、AR-O3 は実装修正（IAM 最小権限化）、AR-O4 は実装修正（構造化ログ導入）

### Q2: 振る舞い変更の許容範囲 — 純粋リファクタリングか、観測された不具合素地の修正も含むか

RE は振る舞いに影響し得る不具合素地を複数観測している: 不正 JSON ボディが 500 になる（API-O8）、フロントのミューテーション失敗時にエラー表示がない（API-O9）、一覧の表示順が不定（BO-O5）、更新/削除が非アトミックで update が upsert になり得る（API-O2）。

a) 外部から観測可能な振る舞いを完全保持する純粋リファクタリングに限定する（上記はすべて現状維持）
b) API コントラクトの後方互換を保てる範囲で不具合素地も修正する（不正 JSON→400、ミューテーションのエラー表示追加、一覧順の保証＝createdAt 降順、条件付き書込によるアトミック化。未使用コード削除はフロント側 `fetchTodo` のみでエンドポイントは維持）
c) API 意味論の変更まで許容する（b に加え PUT→PATCH 化、ページネーションパラメータ追加、未使用エンドポイント GET /api/todos/:id の削除）
d) その他

**Trade Offs:** (a) はリスク最小で「リファクタリング」の定義に忠実だが、500 誤返却や未処理 rejection という明確な品質欠陥を温存する。(b) は既存クライアント（本件では同梱 SPA のみ）を壊さずに品質欠陥を解消でき、修正規模も小さい。(c) は API 契約の破壊的変更を伴い、frontend・テスト・v1 API Design 記述への波及が大きく、「振る舞いを保ったままの品質向上」という確認済みの主軸から逸脱する。

**Recommendation:** **b**。観測された不具合素地は「品質・保守性の向上」の範疇であり、コントラクト互換を保つ限り修正すべき。PATCH 化やエンドポイント削除（c）は本 intent では Out of Scope とし、PUT の部分更新意味論は文書化で対応する（API-P3 の低コスト側）。

[Answer]: b — API コントラクトの後方互換を保てる範囲で不具合素地も修正する（不正 JSON→400、エラー表示追加、一覧順保証、条件付き書込によるアトミック化）

### Q3: 新規 NFR のスコープ — セキュリティ・観測性・検証ゲートをどこまで強化するか

v1 NFR-003 は「認証なし（デモ用途）」。RE は CI/CD 不在（CS-O5）、E2E 不在（CS-O4）、観測性がログのみ（AR-O8）、execute-api 直接アクセス可（AR-O2）を観測している。

a) 現状容認を明文化するに留める（認証なし・直接アクセス容認・CI なしを要件として記録するのみ）
b) 軽量強化: 構造化ログ（SECURITY-03 実装）＋エラースタックのサーバーログ出力（AR-P3）＋ CloudWatch アラーム（AR-P6）＋ CI ゲート（lint / typecheck / test / cdk synth — CS-P3）＋ IAM 最小権限化。WAF・認証・E2E は引き続き Out of Scope
c) 本格強化: b に加えて E2E スモークテスト（CS-P4）＋ execute-api 直接アクセス対策（CloudFront カスタムヘッダ検証 — AR-P2a）＋依存自動更新（Renovate — DEP-P3）
d) その他

**Trade Offs:** (a) はコストゼロだが、確認済み目的③（テスト強化）④（インフラ刷新）に対して成果が薄い。(b) は教材として「運用品質の実装例」を示しつつ、ランニングコスト増はほぼゼロ（アラームのみ微増）。(c) は品質保証が最も厚く、E2E はモック境界では捕捉できない統合不具合（型重複に起因するコントラクト乖離など）を検知できるが、CI 時間と保守対象が増える。WAF は月額コストが発生するためデモ用途では過剰。

**Recommendation:** **c（ただし WAF は除外、直接アクセス対策はカスタムヘッダ検証の軽量実装）**。E2E は data-testid が全要素に整備済みで導入下地があり（CS-O4）、リファクタリングの「振る舞い保持の証明」装置として本 intent と相性が最も良い。認証はデモ用途の前提を維持し Out of Scope を継続する。

[Answer]: c — 本格強化（WAF 除外、直接アクセス対策はカスタムヘッダ検証の軽量実装）。E2E スモーク＋ Renovate 含む。認証はデモ用途前提で Out of Scope 継続

### Q4: 優先順位付けの軸 — RF 要件の P0/P1/P2 をどの基準で割り当てるか

a) リスク優先: セキュリティ・正しさに直結するもの（IAM 最小権限、エラーハンドリング、ドリフト解消）から P0
b) 教材価値優先: AI-DLC 実践教材として見せたい実例（共有パッケージ、CI、構造化ログ）から P0
c) 安全な改修順序を優先: 「壊していないことの証明」を先に固める — P0 = 検証ゲート（CI・テスト・E2E）とコントラクト一元化（shared パッケージ）、P1 = backend / frontend の実装品質修正、P2 = インフラ刷新・依存更新
d) その他

**Trade Offs:** (a) はリスク低減が速いが、回帰検知の網がない状態で本体改修に入るため、リファクタリング自体の安全性が下がる。(b) は教材ストーリーは良いが優先度の根拠が主観的になりやすい。(c) はリファクタリングのセオリー（先にセーフティネット、次に本体、最後に基盤）に沿い、P0 完了後の全変更が検証ゲートで保護される。リスク項目（IAM 等）の着手が P1 以降にずれる点が許容できるかが判断点（本システムは認証なしデモであり、即時の脅威は限定的）。

**Recommendation:** **c**。brownfield リファクタリングでは回帰検知の確立が他のすべての変更の前提条件になる。教材としても「セーフティネット→本体→基盤」の順序自体が AI-DLC の見せ場になる。

[Answer]: c — 安全な改修順序。P0=検証ゲート＋コントラクト一元化、P1=実装品質、P2=インフラ・依存更新
