## 指摘事項（重要度順）

### 高 — refine で修正すべき不整合

**H-1. RF-01 の「3 パッケージ」固定が RF-03 と矛盾する**
- 箇所: §P0 RF-01 要件・受入基準（「3 パッケージの `tsc --noEmit`」「3 パッケージの `vitest run`」）、NFR-006 の Measure
- RF-03 が `@todo-ai-dlc/shared` を新設すると workspace は **4 パッケージ**になる。RF-01 の受入基準を字句どおり満たした CI は shared の typecheck/test を検証対象外にし、コントラクト一元化の要であるパッケージが無検証になる。
- 提案: 「3 パッケージ」を「全 workspace パッケージ」へ変更（RF-01 要件・受入基準・NFR-006 の 3 箇所）。

**H-2. BT-7 の検証手段「E2E」に要件の裏付けがない**
- 箇所: §振る舞い保持要件 BT 表の BT-7 行（検証手段: backend unit + **E2E**）vs RF-02（E2E の範囲は **BT-1〜BT-5** のみ）
- BT-7（`/api/health`）を E2E で検証する要件はどこにも存在しない。RF-16（直接アクセス 403 化）は API 側のミドルウェアで実装される可能性が高く、health エンドポイントが意図せず 403 化する回帰は単体テストの境界をすり抜けやすい — E2E での 1 アサーション追加は技術的に妥当かつ安価。
- 提案: RF-02 の範囲を「BT-1〜BT-5 + BT-7（health 200 の 1 アサーション）」へ拡張するか、BT-7 行の検証手段から E2E を削除して整合させる。前者を推奨。

**H-3. RF-07 のエッジケース: 400/404 の優先順位が変わる（BP-1 の承認リスト外）**
- 箇所: RF-07 受入基準「既存の 200/404 応答が不変」、BP-1 の許容変更リスト
- 現実装の update は `findById`（404 判定）→ zod 検証（400 判定）の順（api-documentation.md「Internal Interfaces」どおり）。条件付き書込化すると順序が必然的に **検証 → 書込（404）** へ反転し、「不正ボディ かつ 存在しない id」のリクエストへの応答が **404 → 400 に変わる**。これは外部から観測可能な振る舞い変更だが、Q2=b の承認 4 件にも BP-1 の許容リストにも含まれていない。
- 提案: BP-1 の許容変更に「RF-07 に伴う 400/404 優先順位の変更（不正ボディ + 不存在 id の複合ケースに限る）」を明記するか、RF-07 の受入基準でこの複合ケースの期待値（400 を正とする等）を確定する。設計ステージへの先送りでもよいが、要件として「未決定である」ことの明示は必要。

**H-4. RF-16 の受入基準にローカル開発・E2E 経路の動作保証がない**
- 箇所: RF-16 受入基準（「CloudFront 経由の全 BT が不変」のみ。RF-12 が「本番 + ローカル開発の両方」を明記しているのと非対称）
- カスタムヘッダ検証は backend コード（Lambda/dev.ts 共通の Hono ミドルウェア）に入る公算が大きい。docker-compose / Vite proxy 経路には CloudFront が存在しないため、検証を環境で無効化するか compose 側でヘッダを注入しない限り、**RF-02 の E2E と全ローカル開発が 403 で壊れる**。
- 提案: RF-16 の受入基準に「docker-compose / Vite proxy のローカル経路で全 BT が動作する（E2E がパスする）」を追加。
- 補足（設計向けメモ）: 「ヘッダ値はハードコードされない」は満たせるが、CloudFront origin custom header の値は synth したテンプレート・Lambda 環境変数には現れ得る（Secrets Manager 動的参照を使っても CloudFront 側の解決値はテンプレート閲覧者に見える）。認証なしデモでの軽量実装としては許容範囲だが、「Secret/パラメータ管理」が何を守るか（リポジトリへの平文コミット防止まで）を設計時に定義すること。

### 中 — 受入基準の字句修正・スコープ明確化を推奨

**M-1. RF-12「CORS 設定がリポジトリ内 1 箇所のみに存在する」は撤去案と衝突する**
- AR-P1（原典）は「Hono `cors()` は削除（Vite proxy 経由なら不要）」を選択肢として含む。設計が「両方撤去（0 箇所）」を選ぶと、技術的に最善でも受入基準を字句上満たせない。「高々 1 箇所（二重定義が解消されている）」へ修正を推奨。
- 関連: CORS ヘッダの消失は cross-origin クライアントには観測可能な変更だが、BP-1 の許容リストに含まれていない。A-1（既知クライアントは同梱 SPA のみ）を根拠に許容する旨を BP-1 か RF-12 に一言追記すると閉じる。

**M-2. NFR-001 のフロント初回ロード 3 秒は RF-15 後も未計測のまま**
- §v1 検証結果表は「NFR-001 未検証 → RF-15 で計測可能化」とするが、RF-15 のアラーム 4 種（Lambda Errors/Duration、API GW 5xx/Latency）が計測可能にするのは **API 側 500ms のみ**。フロント 3 秒側の計測手段（RUM 等）は OOS-9 で明示的に除外されている。
- 提案: 検証結果表と NFR-001 の Measure に「フロント初回ロード 3 秒は本 intent では計測対象外（OOS-9 と整合）」と明記。現状の記述は RF-15 の達成範囲を過大に見せる。

**M-3. RF-15 のアラーム閾値 500ms はコールドスタートで誤報する**
- NFR-001 は「コールドスタート除く」だが、API GW Latency メトリクスはコールドスタート時の統合レイテンシを**含む**（Lambda Duration メトリクスは init を含まないので影響小）。Latency 閾値 500ms を単純な平均/単一データポイントで張ると恒常的なノイズ源になる。
- 提案: 受入基準は現状のまま検証可能だが、「閾値が 500ms と整合」に「統計（p95/p99）と評価期間は設計ステージで NFR-001 の『コールドスタート除く』条件を満たすよう決定する」と添えると、設計が naive な平均値アラームに落ちるのを防げる。

**M-4. RF-06「安定表示」には同時刻 tie の扱いが必要**
- createdAt は ISO 8601 ミリ秒精度で、同一ミリ秒の複数アイテムは createdAt 単独ソートでは順序が不定のまま（「安定表示」を満たさない）。ULID は辞書順 = 時系列なので、第 2 ソートキー（id 降順）で決定的にできる（BO-P4 の備考どおり）。設計判断事項として受入基準か備考に一言追記を推奨。

### 低 — 設計ステージへの申し送りで足りる

**L-1. RF-03「リテラルが共有定数の 1 箇所のみ」のスコープ**: テストコードの境界値（`"a".repeat(201)` 等）が字句上の違反になり得る。「プロダクションコード内で 1 箇所」とスコープを限定するか、テストは共有定数から導出する方針を設計で決める。
**L-2. RF-01 の `pnpm audit` はゲートとして不安定になり得る**: 上流の修正不能 advisory で CI が恒常 fail する既知パターン。severity 閾値（例: `--audit-level=high`）の設定を設計ステージで決定する旨を備考に。
**L-3. RF-18 の補足 2 点**: ① aws-cdk CLI はバージョン体系が aws-cdk-lib と分離済み（2.1000 系）であり、「現行 2.x へ」は CLI/lib で別番号になる（受入基準「完全固定で宣言」は引き続き検証可能）。② Vite 7 は Node エンジン要件が上がる（^20.19 / >=22.12）— `node:20-slim` の追従確認を移行手順に含める。また TS-O5（lockfile に react 19.2.4/19.2.14 併存）の解消を RF-18 に期待するなら一言あるとよい。
**L-4. トレーサビリティの微細な補強（誤りではない）**: RF-10 の出典に TS-O6 を併記可能（TS-P4 の原典対応は「TS-O6, AR-O4」）。RF-18 の出典に CS-O9（Biome 1.9 観測）を併記可能。A-1 の根拠として AR-O10（CSP `connect-src 'self'` が同一オリジン前提と整合）を補強材料に使える。

## 確認済み — 問題なしの領域

1. **トレーサビリティ（全件原典照合済み・誤引用ゼロ）**: RF-01〜22 の出典に挙がる全 Observation/Proposal ID（BO-O1〜O6/BO-P1〜P5、AR-O1〜O10/AR-P1〜P7、CS-O1〜O9/CS-P1〜P6、API-O1〜O9/API-P1〜P6、TS-O1〜O6/TS-P1〜P5、DEP-O1〜O5/DEP-P1〜P3）が原典に実在し、内容の対応も正確。RF-14 の Proposal 欄「—」も正しい（AR-O3 に対する提案は RE 側に存在しない。観測のみから Q1=c で要件化した経緯が誠実に記録されている）。
2. **Proposal の処理網羅性**: RE の全 29 Proposals が採用（RF 化）・一部採用・不採用のいずれかに漏れなく対応付き。不採用 9 件の理由は原典のトレードオフ記述と技術的に一致する（特に TS-P2: zod v4 の `flatten()` 廃止が 400 レスポンス形状＝API 契約へ波及するため Q2=b と衝突、は正確）。
3. **BP-1 の数値**: 既存 45 テストケース（backend 21 = handler 14 + repository 7 / frontend 17 = App 3 + Form 5 + Item 6 + List 3 / infrastructure 7）は code-structure.md の実数と一致。
4. **RF-14 と RF-07 の干渉なし（検証済み)**: 条件付き書込（`ConditionExpression`）は UpdateItem/DeleteItem 権限の範囲内で動作し、追加アクション（`ConditionCheckItem` はトランザクション専用）を要しない。v1 設計の 5 アクション集合（GetItem/PutItem/UpdateItem/DeleteItem/Scan）は RF-07 適用後も必要十分。
5. **P0/P1/P2 と Q4=c の整合**: P0×3 / P1×10 / P2×9 の割当は「検証ゲート → 本体 → 基盤」の軸に正しく従う。IAM（RF-14）を P2 に置くトレードオフは questions.md Q4 で人間が明示的に受け入れ済み。RF-18（メジャー更新）を P0 確立後に置く順序は、テスト資産を Vitest 2→3 で二度更新するコストより回帰安全性を優先する判断として技術的に妥当。
6. **依存順序の技術的妥当性**: RF-03（shared）→ RF-04〜07（handler/frontend 改修）の順序は手戻りを最小化する。RF-17 を RF-18 と同時実施する判断も妥当（なお `pointInTimeRecoverySpecification` は固定中の 2.177.0 でも利用可能なため、RF-18 を待たず単独実施も技術的には可能 — 制約ではなく効率の問題として正しく書かれている）。infrastructure → backend/frontend のパス結合（DEP-O1）は RF-01 の `cdk synth` ゲートと RF-20 のスクリプト化で正しくカバーされる。
7. **受入基準の検証可能性**: 上記 H/M 指摘の字句問題を除き、22 件すべてが grep / unit test / Template assertion / E2E / 文書照合のいずれかで pass/fail 判定可能。検証手段の割当（例: RF-14・RF-15 を infrastructure の Template assertion で検証）は実装イメージと一致しており実行可能。
8. **Assumptions / OOS の技術的整合**: A-5/A-6（単一環境・データ移行なし）と OOS-8（AR-P4 不採用）の整合、A-7（E2E はローカルのみ）と RF-02 の整合、OOS-6（zod v4)/OOS-7（OpenAPI）の将来 intent への送り方は、RE 著者として意図どおりの解釈。

## 結論
